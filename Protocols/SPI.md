# SPI (Serial Peripheral Interface)

## Overview
SPI is a synchronous, full-duplex serial communication protocol designed for short-distance communication between microcontrollers and peripherals. Developed by Motorola, it's widely used for fast data transfer between devices on a circuit board.

## Technical Specifications

| Parameter | Value |
|-----------|-------|
| **Wires Required** | 4 (MOSI, MISO, SCK, CS) |
| **Transmission** | Synchronous, Full-Duplex |
| **Clock Speed** | 1 MHz - 100 MHz+ (depends on device) |
| **Distance** | ~1 meter (PCB traces) |
| **Voltage Levels** | 5V (TTL), 3.3V, 1.8V |
| **Devices per bus** | Unlimited (via CS lines) |
| **Data Format** | 8-bit, 16-bit, 32-bit programmable |
| **Latency** | Very low (microseconds) |

## Pin Configuration & Signal Description

```
SPI Connector (6-pin standard):
┌─────────────────┐
│  1   2   3   4  │
│ MOSI SCK MISO CS│
│  5   6          │
│ GND +5V         │
└─────────────────┘

Signal Lines:

MOSI (Master Out, Slave In):
- Output from Master
- Input to Slave(s)
- Transfers data from Master→Slave

MISO (Master In, Slave Out):
- Input to Master
- Output from Slave
- Transfers data from Slave→Master

SCK (Serial Clock):
- Generated by Master
- Synchronizes data transmission
- Both MOSI and MISO data change with clock

CS (Chip Select):
- Active LOW (typically)
- Selects which slave is active
- Multiple slaves use multiple CS lines
- Pulled HIGH when not selected
```

## How SPI Works

### Basic Communication Steps:

1. **Slave Selection (CS goes LOW)**
   ```
   Master pulls CS LOW
   Selects target slave for communication
   Slave detects CS=LOW and becomes active
   ```

2. **Clock Generation**
   ```
   Master generates SCK clock
   Both Master and Slave synchronize to SCK
   Data changes on one edge, sampled on opposite edge
   ```

3. **Data Transmission (Full-Duplex)**
   ```
   Master shifts out data on MOSI (MSB first typically)
   Slave shifts out data on MISO simultaneously
   Each clock pulse shifts 1 bit
   ```

4. **Byte Completion**
   ```
   After 8 clock pulses, full byte transmitted and received
   Shift register full on both sides
   Data available in receive buffer
   ```

5. **Slave Deselection (CS goes HIGH)**
   ```
   Master releases CS (goes HIGH)
   Slave detects CS=HIGH and becomes inactive
   Slave MISO line goes to high-impedance
   ```

### Detailed Transaction Example:

```cpp
// Transmit 0xA5 (10100101), Receive unknown byte

Initialization:
Master: SCK=0, CS=1, MOSI=0
Slave:  SCK=0, MISO=Z (high-impedance)

Slave select:
Master: CS=0  ← Slave activated
Slave:  Ready to shift

Bit 0 (MSB = 1):
Master: Shifts out 1 on MOSI
         MOSI=1 (output)
         SCK: 0→1 (clock pulse)
Slave:  Reads MOSI when SCK rises
         Shifts in and out

Bit 1 (0):
Master: MOSI=0
         SCK: 1→0→1
         Reads MISO when SCK rises

...continues for 8 bits...

End of byte:
Master: SCK returns to 0
         CS: 0→1 (slave deselected)
Slave:  MISO goes to high-impedance

Result: Both transmitted and received 8-bit values
```

## Interview Questions & Answers

### BEGINNER LEVEL

**Q1: What does SPI stand for?**
A: SPI stands for Serial Peripheral Interface. It's a synchronous, full-duplex serial communication protocol used for high-speed communication between a master device (usually a microcontroller) and slave devices (sensors, memory, displays, etc.).

**Q2: What are the four main signals in SPI?**
A:
- **MOSI** (Master Out, Slave In): Data from Master to Slave
- **MISO** (Master In, Slave Out): Data from Slave to Master
- **SCK** (Serial Clock): Clock signal generated by Master
- **CS** (Chip Select): Active LOW signal to select specific slave

**Q3: What's the difference between SPI and I2C?**
A:
| Feature | SPI | I2C |
|---------|-----|-----|
| Wires | 4+ | 2 |
| Speed | Much faster (~10 MHz) | Slower (100-400 kHz) |
| Multi-slave | Yes (separate CS) | Yes (addressing) |
| Full-duplex | Yes | No |
| Distance | ~1 meter | ~1 meter |
| Complexity | Medium | More complex |

**Q4: Is SPI synchronous or asynchronous?**
A: SPI is **synchronous**. The master generates a clock signal (SCK) that synchronizes all data transfers. Both Master and Slave use this clock to know exactly when to shift bits in/out.

**Q5: How many slaves can SPI support?**
A: Theoretically unlimited. Each slave needs a separate CS (Chip Select) line. With N slaves, you need N CS lines (plus 3 shared lines: MOSI, MISO, SCK).

**Q6: What is the CS (Chip Select) signal used for?**
A: CS signal (active LOW):
- Activates selected slave
- Allows multiple slaves on same bus
- Slave ignores data when CS=HIGH
- Master controls CS lines (GPIO pins)
- Typical sequence: Pull CS LOW → Transfer → Release CS HIGH

**Q7: What are SPI modes?**
A: Four SPI modes based on Clock Polarity (CPOL) and Clock Phase (CPHA):
- **Mode 0** (CPOL=0, CPHA=0): Clock idles LOW, data sampled on rise
- **Mode 1** (CPOL=0, CPHA=1): Clock idles LOW, data sampled on fall
- **Mode 2** (CPOL=1, CPHA=0): Clock idles HIGH, data sampled on fall
- **Mode 3** (CPOL=1, CPHA=1): Clock idles HIGH, data sampled on rise

Most common: **Mode 0** and **Mode 3**

```
- CPOL=0: Clock idles LOW
- CPOL=1: Clock idles HIGH

        Good for devices that need more settling time

        Less common variant

        Alternative to Mode 0
```

**Q10: What is the difference between full-duplex and half-duplex?**
A:
```
Full-Duplex (SPI):
Master                          Slave
MOSI: sends data  ──────────→   Receives
MISO: receives ←──────────────  Sends data
SCK:  clock pulse synchronizes both

Timing:
SCK:  ┌─┐ ┌─┐ ┌─┐ ┌─┐
      └─┘ └─┘ └─┘ └─┘
MOSI: ─D7──D6──D5──D4───
MISO: ─D7──D6──D5──D4───
(Simultaneous transmission and reception)

Half-Duplex (I2C):
Only one line per direction, cannot transmit and receive simultaneously
Time-divided access (turnaround delay)

SPI advantage: Full-duplex allows simultaneous transfers
```

**Q11: Explain the difference between hardware and software SPI.**
A:
```
Hardware SPI:
□ Uses dedicated SPI peripheral on microcontroller
□ Dedicated pins: MOSI, MISO, SCK (cannot reassign)
□ Dedicated shift register logic
□ Clock generation in hardware
□ Fast: Automated, no CPU overhead
□ Sample: AVR, ARM, PIC

Advantages:
+ Very fast (MHz speeds)
+ Low CPU usage
+ Precise timing
+ Built-in DMA support often available

Disadvantages:
- Fixed pins
- Limited to one concurrent transfer

Software SPI (Bit-Banging):
□ Any GPIO pins can be used
□ Manual bit shifting in software
□ CPU-intensive
□ Slower (kHz speeds typical)
□ More flexible

Example (Software SPI):
void spi_write_byte(uint8_t data) {
    for (int i = 7; i >= 0; i--) {
        // Set MOSI to bit
        digitalWrite(MOSI, (data >> i) & 1);
        
        // Clock pulse
        digitalWrite(SCK, HIGH);
        delayMicroseconds(10);
        digitalWrite(SCK, LOW);
    }
}
```

**Q12: What is the maximum SPI clock frequency?**
A:
```
Depends on multiple factors:

1. Device Specifications:
   - Slave maximum speed (check datasheet)
   - Master SPI peripheral max
   - Example: STM32 can do 72 MHz SPI
   - Example: ATmega328p can do ~10 MHz

2. Physical Limitations:
   - Cable length and quality
   - PCB trace impedance
   - Circuit board layout
   - EMI/RFI noise
   - Short PCB traces: 10-50 MHz typically OK
   - Long wires: 1-5 MHz

3. Propagation Delay:
   Clock period must be >> propagation delay
   Setup time: Data must be stable before clock
   Hold time: Data must remain stable after clock

Practical Maximum Speeds:
□ Microcontroller to on-board IC: 25-50 MHz
□ Microcontroller to off-board device: 5-25 MHz
□ Long cable shielded: <5 MHz
□ Bit-banged software SPI: <1 MHz

Formula consideration:
Period = 1/Frequency
Example: 10 MHz = 100 ns period
Propagation must be <10 ns typically
```

**Q13: How do you detect SPI communication errors?**
A:
```
SPI doesn't have built-in error detection (unlike I2C with ACK/NACK)
Errors must be handled at higher protocol level:

Common Errors:

1. Data Corruption:
   Cause: EMI/RFI interference
   Solution: Add shielded cable, proper grounding,
             reduce clock speed, add filtering

2. Timing Errors:
   Cause: Clock mismatch, wrong mode selection
   Solution: Verify CPOL, CPHA, verify clock frequency,
             check setup/hold times, use oscilloscope

3. Software Integration:
   Cause: Incorrect bit ordering, wrong slave selected
   Solution: Verify MSB/LSB setting, check CS timing,
             test with known data patterns

4. Packet Loss:
   No inherent detection mechanism
   Solution: Add checksum at protocol level
            Add time-out detection
            Add ACK/NAK handshake in firmware

Example Error Detection:
typedef struct {
    uint8_t data[32];
    uint8_t checksum;
} spi_packet_t;

uint8_t calc_checksum(uint8_t *data, uint8_t len) {
    uint8_t sum = 0;
    for(int i=0; i<len; i++) sum += data[i];
    return ~sum;  // One's complement
}

bool verify_packet(spi_packet_t *pkt) {
    uint8_t calc = calc_checksum(pkt->data, 32);
    return (calc == pkt->checksum);
}
```

### ADVANCED LEVEL

**Q14: Design an SPI communication system to interface with 3 devices: EEPROM, LCD, and SD Card.**
A:
```cpp
#include <avr/io.h>

// Device Addresses (CS pins)
#define CS_EEPROM PB0  // Device Select for EEPROM
#define CS_LCD    PB1  // Device Select for LCD
#define CS_SD     PB2  // Device Select for SD Card

// SPI Configuration
void spi_init() {
    // Set SS, MOSI, SCK as outputs; MISO as input
    DDRB |= (1 << PB4) | (1 << PB5) | (1 << PB7);
    DDRB &= ~(1 << PB6);
    
    // Set CS pins as outputs (HIGH initially)
    DDRB |= (1 << CS_EEPROM) | (1 << CS_LCD) | (1 << CS_SD);
    PORTB |= (1 << CS_EEPROM) | (1 << CS_LCD) | (1 << CS_SD);
    
    // Enable SPI, Master mode, Clock = F_CPU/4
    SPCR = (1 << SPE) | (1 << MSTR) | (1 << SPR0);
}

// SPI Byte Transfer
uint8_t spi_transfer(uint8_t data) {
    SPDR = data;
    while (!(SPSR & (1 << SPIF)));
    return SPDR;
}

// EEPROM Interface (AT25C256)
void eeprom_select() {
    PORTB &= ~(1 << CS_EEPROM);  // CS LOW
}

void eeprom_deselect() {
    PORTB |= (1 << CS_EEPROM);   // CS HIGH
}

void eeprom_write(uint16_t addr, uint8_t data) {
    eeprom_select();
    spi_transfer(0x02);           // WRITE command
    spi_transfer((addr >> 8) & 0xFF);  // Address High
    spi_transfer(addr & 0xFF);         // Address Low
    spi_transfer(data);            // Data
    eeprom_deselect();
}

uint8_t eeprom_read(uint16_t addr) {
    uint8_t data;
    eeprom_select();
    spi_transfer(0x03);           // READ command
    spi_transfer((addr >> 8) & 0xFF);
    spi_transfer(addr & 0xFF);
    data = spi_transfer(0xFF);    // Dummy read, receive data
    eeprom_deselect();
    return data;
}

// LCD Interface (ILI9341)
void lcd_select() {
    PORTB &= ~(1 << CS_LCD);
}

void lcd_deselect() {
    PORTB |= (1 << CS_LCD);
}

void lcd_write_command(uint8_t cmd) {
    lcd_select();
    spi_transfer(cmd);
    lcd_deselect();
}

void lcd_write_data(uint8_t data) {
    lcd_select();
    spi_transfer(data);
    lcd_deselect();
}

// SD Card Interface
void sd_select() {
    PORTB &= ~(1 << CS_SD);
}

void sd_deselect() {
    PORTB |= (1 << CS_SD);
}

int sd_initialize() {
    uint8_t i;
    // Send 80 dummy clock pulses before initialization
    sd_deselect();
    for (i = 0; i < 10; i++) {
        spi_transfer(0xFF);
    }
    
    sd_select();
    // Send CMD0 (Reset)
    spi_transfer(0x40);  // CMD0
    spi_transfer(0x00);
    spi_transfer(0x00);
    spi_transfer(0x00);
    spi_transfer(0x00);
    spi_transfer(0x95);  // CRC
    
    uint8_t response = spi_transfer(0xFF);
    sd_deselect();
    
    return (response == 0x01) ? 1 : 0;  // 0x01 = Idle state
}

// Main Application
int main() {
    spi_init();
    
    // Initialize devices
    eeprom_select();
    _delay_ms(10);
    eeprom_deselect();
    
    lcd_write_command(0x01);  // Software reset
    _delay_ms(20);
    
    if (sd_initialize() == 1) {
        // SD card ready
    }
    
    // Test multidevice communication
    uint8_t data = 0xAA;
    eeprom_write(0x0000, data);
    _delay_ms(10);
    
    data = eeprom_read(0x0000);
    
    while(1) {
        // Application loop
    }
    
    return 0;
}
```

**Q15: Implement SPI with DMA (Direct Memory Access) for high-speed data transfer.**
A:
```cpp
// STM32 SPI with DMA Example

#include "stm32f4xx.h"

// Setup SPI1 with DMA
void spi1_dma_init() {
    // Enable clocks
    RCC->APB2ENR |= RCC_APB2ENR_SPI1EN;
    RCC->AHB1ENR |= RCC_AHB1ENR_DMA2EN;
    
    // Configure SPI1
    SPI1->CR1 = 0;
    SPI1->CR1 |= SPI_CR1_MSTR;           // Master mode
    SPI1->CR1 |= SPI_CR1_SSM;            // Software SS
    SPI1->CR1 |= SPI_CR1_SSI;
    SPI1->CR1 |= (2 << SPI_CR1_BR_Pos);  // Baud rate divider
    SPI1->CR2 |= SPI_CR2_TXDMAEN;        // Enable TX DMA
    SPI1->CR2 |= SPI_CR2_RXDMAEN;        // Enable RX DMA
    SPI1->CR1 |= SPI_CR1_SPE;            // Enable SPI
    
    // Configure DMA
    // TX: DMA2 Stream 3
    DMA2_Stream3->CR = 0;
    DMA2_Stream3->CR |= (3 << DMA_SxCR_CHSEL_Pos);  // Channel 3
    DMA2_Stream3->CR |= DMA_SxCR_DIR_0;  // Memory to Peripheral
    DMA2_Stream3->CR |= DMA_SxCR_MINC;   // Memory increment
    DMA2_Stream3->CR |= (1 << DMA_SxCR_PSIZE_Pos); // 8-bit
    DMA2_Stream3->CR |= (1 << DMA_SxCR_MSIZE_Pos);
    
    // RX: DMA2 Stream 0
    DMA2_Stream0->CR = 0;
    DMA2_Stream0->CR |= (3 << DMA_SxCR_CHSEL_Pos);
    DMA2_Stream0->CR |= DMA_SxCR_MINC;
    DMA2_Stream0->CR |= (1 << DMA_SxCR_PSIZE_Pos);
    DMA2_Stream0->CR |= (1 << DMA_SxCR_MSIZE_Pos);
}

// SPI Transfer with DMA
void spi_dma_transfer(uint8_t *tx_buf, uint8_t *rx_buf, uint16_t len) {
    // Configure TX DMA
    DMA2_Stream3->NDTR = len;
    DMA2_Stream3->PAR = (uint32_t)&(SPI1->DR);
    DMA2_Stream3->M0AR = (uint32_t)tx_buf;
    
    // Configure RX DMA
    DMA2_Stream0->NDTR = len;
    DMA2_Stream0->PAR = (uint32_t)&(SPI1->DR);
    DMA2_Stream0->M0AR = (uint32_t)rx_buf;
    
    // Enable DMA streams
    DMA2_Stream3->CR |= DMA_SxCR_EN;
    DMA2_Stream0->CR |= DMA_SxCR_EN;
    
    // Wait for completion
    while ((DMA2->LISR & DMA_LISR_TCIF0) == 0);  // RX complete
    while ((DMA2->LISR & DMA_LISR_TCIF3) == 0);  // TX complete
    
    // Clear flags
    DMA2->LIFCR = DMA_LIFCR_CTCIF0 | DMA_LIFCR_CTCIF3;
}

// High-speed block transfer
void high_speed_block_write(uint8_t *data, uint32_t length) {
    uint8_t rx_dummy[MAX_BUFFER];
    
    // Chip select
    GPIOC->BSRR = (1 << 6);  // CS LOW
    
    // Start DMA transfer
    spi_dma_transfer(data, rx_dummy, length);
    
    // Chip deselect
    GPIOC->BSRR = (1 << (6 + 16));  // CS HIGH
}

Benefits of DMA SPI:
+ CPU not involved in bit shifting
+ Transfer happens in parallel
+ Can transfer blocks at full SPI speed
+ Reduces CPU overhead
+ Ideal for SD card, file systems, displays
+Typical throughput: 10+ MB/s possible (vs ~100KB/s with software)
```

**Q16: Analyze SPI timing and calculate maximum clock frequency.**
A:
```
SPI Timing Analysis:

Key Timing Parameters:
1. Setup time (tSU): Time data must be stable before clock edge
2. Hold time (tH): Time data must remain after clock edge
3. Propagation delay (tPD): Chip output to next receiver
4. Clock frequency: 1/Period

Master Timing Constraints:
                    ┌─────┐
SCK:                └────ld┘
      ├─tSU─┤      ├─tH─┤
MOSI: ─D[n]─────────D[n+1]─

Slave Timing:
                    ┌─────┐
SCK:                └─────┘
Slave tPD: 0-30ns ▶─┐
MISO:               D[n] is valid here

Calculation:
Period_min = 2 × (tPD_max + tSU_max) + margin

Example:
- Device setup time: 20ns
- Propagation delay: 15ns
- Margin: 10ns
Period_min = 2 × (20 + 15) + 10 = 80ns
Max frequency = 1/80ns = 12.5 MHz

Practical considerations:
• Datasheet max frequency often conservative
• PCB trace length affects propagation
• Temperature affects timing
• Multiple slaves increase load
• Ground plane quality affects speed
• Decoupling capacitors essential

Test procedure:
1. Set frequency according to slowest device
2. Use oscilloscope to verify setup/hold
3. Start at slow frequency, gradually increase
4. Monitor MISO line for violations
5. Add timing tolerance ~20% below max calculated
```

**Q17: Design error detection and correction for SPI.**
A:
```cpp
// Advanced SPI Protocol with Error Detection

#define FRAME_SYNC 0xAA
#define FRAME_END  0x55
#define MAX_PAYLOAD 64

typedef struct {
    uint8_t sync;           // Start marker
    uint8_t cmd;            // Command
    uint8_t length;         // Data length
    uint8_t payload[MAX_PAYLOAD];
    uint8_t crc8;           // CRC-8
    uint8_t end;            // End marker
} spi_frame_t;

// CRC-8 calculation (Polynomial: x^8 + x^7 + x^6 + x^4 + x^2 + 1)
uint8_t crc8_calculate(uint8_t *data, uint8_t len) {
    uint8_t crc = 0xFF;
    for (int i = 0; i < len; i++) {
        crc ^= data[i];
        for (int j = 0; j < 8; j++) {
            if (crc & 0x80) {
                crc = (crc << 1) ^ 0x07;
            } else {
                crc = crc << 1;
            }
        }
    }
    return crc;
}

// Transmit frame with error detection
bool spi_send_frame(spi_frame_t *frame) {
    uint8_t *data = (uint8_t *)frame;
    uint16_t frame_len = sizeof(spi_frame_t);
    
    // Set markers
    frame->sync = FRAME_SYNC;
    frame->end = FRAME_END;
    
    // Calculate CRC (all except CRC field)
    frame->crc8 = crc8_calculate(
        &frame->cmd, 
        frame->length + 2  // cmd + length + payload
    );
    
    // Transmit frame
    for (int i = 0; i < frame_len; i++) {
        spi_transfer(data[i]);
    }
    
    return true;
}

// Receive frame with error detection
bool spi_receive_frame(spi_frame_t *frame) {
    uint8_t *data = (uint8_t *)frame;
    uint16_t frame_len = sizeof(spi_frame_t);
    
    // Receive all bytes
    for (int i = 0; i < frame_len; i++) {
        data[i] = spi_transfer(0xFF);
    }
    
    // Verify sync marker
    if (frame->sync != FRAME_SYNC) {
        return false;  // Sync error
    }
    
    // Verify CRC
    uint8_t calc_crc = crc8_calculate(
        &frame->cmd, 
        frame->length + 2
    );
    if (calc_crc != frame->crc8) {
        return false;  // CRC mismatch
    }
    
    // Verify end marker
    if (frame->end != FRAME_END) {
        return false;  // Frame corruption
    }
    
    return true;
}

// Request-Response with retry logic
#define MAX_RETRIES 3

bool spi_reliable_transfer(spi_frame_t *tx, spi_frame_t *rx) {
    for (int attempt = 0; attempt < MAX_RETRIES; attempt++) {
        if (!spi_send_frame(tx)) {
            continue;  // Send error
        }
        
        _delay_ms(100);  // Wait for response
        
        if (spi_receive_frame(rx)) {
            return true;  // Success!
        }
        
        _delay_ms(50);   // Backoff before retry
    }
    
    return false;  // Failed after all retries
}

// Example usage
int main() {
    spi_init();
    
    spi_frame_t tx_frame = {0};
    spi_frame_t rx_frame = {0};
    
    // Prepare transmit frame
    tx_frame.cmd = 0x10;  // Read command
    tx_frame.length = 2;
    tx_frame.payload[0] = 0x00;  // Address High
    tx_frame.payload[1] = 0x10;  // Address Low
    
    // Send with reliability
    if (spi_reliable_transfer(&tx_frame, &rx_frame)) {
        // Process received data
        uint16_t value = (rx_frame.payload[0] << 8) | rx_frame.payload[1];
    } else {
        // Communication failed after retries
        // Handle error
    }
    
    return 0;
}
```

## Advantages & Disadvantages

### Advantages ✓
- Very fast (MHz to 100+ MHz possible)
- Full-duplex communication
- Simple master-slave architecture
- Easy to implement
- No addressing complexity
- Good for short distances
- Low power at high speeds
- Widely supported
- Excellent for real-time data

### Disadvantages ✗
- Requires 4+ wires (vs I2C's 2)
- No built-in error detection
- No acknowledgment mechanism
- Requires separate CS for each slave
- More complex multi-master scenarios
- No standard for higher-level protocol
- Slave must be active (cannot go to low-power)
- Limited distance (~1 meter typical)

## Troubleshooting Guide

| Issue | Cause | Solution |
|-------|-------|----------|
| No data received | CS not toggled | Verify CS timing |
| Corrupted data | Wrong SPI mode | Check CPOL, CPHA |
| Noise on lines | Poor grounding | Improve ground plane |
| Slow transfer | Wrong divider | Recalculate clock divider |
| Device not responding | CS always HIGH | Check CS line logic |
| Intermittent errors | Setup/hold violation | Reduce clock speed |
| Mutual interference | Multiple devices active | Verify CS isolation |
