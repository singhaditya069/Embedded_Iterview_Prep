# Communication Protocols - Comprehensive Q&A

## Serial Communication Protocols

### Q1: UART (Universal Asynchronous Receiver Transmitter)
**Answer:**
```
UART Characteristics:
├─ Type: Asynchronous serial
├─ Speed: 9600, 19200, 115200 baud (common)
├─ Wires: TX, RX (simplex), sometimes RTS/CTS (flow control)
├─ Distance: Up to 15m (industrial up to 100m)
├─ Advantages: Simple, widely supported
└─ Disadvantages: Slow, no clock sync, individual bits can error

UART Frame Structure:
┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
│1 │D0│D1│D2│D3│D4│D5│D6│D7│ P│S│ Start bits: 1
│  │  │  │  │  │  │  │  │  │\ │P│ Data bits: 8
│  │  │  │  │  │  │  │  │  │ \│ │ Parity: Optional
│  │  │  │  │  │  │  │  │  │  \│ Stop bits: 1-2
└──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
 1Start 8Data Parity Stop

Total: 10 bits for 8 data bits = 80% efficiency
At 115200 baud ≈ 11.5 KB/sec theoretical max
```

**Configuration for STM32:**
```c
#include "usart.h"

void init_uart(uint32_t baud) {
    // UART configuration
    #define UART_BASE 0x40011000  // USART1
    
    // Baud rate calculation
    // BRR = fclk / (16 * baud)
    // For fclk = 168MHz, baud = 115200:
    // BRR = 168000000 / (16 * 115200) = 91
    
    uint16_t brr = (SystemCoreClock / (16 * baud));
    
    // Configure
    volatile uint32_t *USART_BRR = (volatile uint32_t *)(UART_BASE + 0x08);
    *USART_BRR = brr;
    
    // Enable TX, RX, UART
    volatile uint32_t *USART_CR1 = (volatile uint32_t *)(UART_BASE + 0x00);
    *USART_CR1 = (1 << 3) | (1 << 2) | (1 << 13);  // TE | RE | UE
}

// Transmit character
void uart_putchar(char c) {
    volatile uint32_t *USART_DR = (volatile uint32_t *)(0x40011000 + 0x04);
    volatile uint32_t *USART_SR = (volatile uint32_t *)(0x40011000 + 0x00);
    
    while (!((*USART_SR) & (1 << 7))) { }  // Wait for empty
    *USART_DR = c;
}

// Receive character
char uart_getchar(void) {
    volatile uint32_t *USART_DR = (volatile uint32_t *)(0x40011000 + 0x04);
    volatile uint32_t *USART_SR = (volatile uint32_t *)(0x40011000 + 0x00);
    
    while (!((*USART_SR) & (1 << 5))) { }  // Wait for data
    return (char)(*USART_DR);
}

// Typical usage: Printf redirect
int _write(int file, char *ptr, int len) {
    for (int i = 0; i < len; i++) {
        uart_putchar(ptr[i]);
    }
    return len;
}
```

### Q2: SPI (Serial Peripheral Interface)
**Answer:**
```
SPI Characteristics:
├─ Type: Synchronous serial (clock-driven)
├─ Speed: 1-50 MHz typical (some up to 100+ MHz)
├─ Wires: MOSI, MISO, SCK, CS (4 wires minimum)
├─ Distance: Short distance (< 1m typically)
├─ Advantages: Fast, full-duplex
└─ Disadvantages: More pins, no built-in addressing

SPI Electrical:
┌──────────┐                ┌──────────┐
│  Master  │                │  Slave   │
│ (MCU)    │                │  (Device)│
│          │                │          │
├─ MOSI ───┼────────────────┼─> MOSI   │
├─ MISO ◄──┼────────────────┼─ MISO    │
├─ SCK  ───┼────────────────┼─> SCK    │  Synchronized
├─ CS   ───┼────────────────┼─> CS     │  by clock
└──────────┘                └──────────┘

MOSI: Master Out, Slave In (data from master)
MISO: Master In, Slave Out (data from slave)
SCK: Serial Clock (generated by master)
CS: Chip Select (active low, selects slave)
```

**Clock Modes (CPOL, CPHA):**
```
Mode 0: CPOL=0, CPHA=0 (clock idle low, sample on rising edge)
Mode 1: CPOL=0, CPHA=1 (clock idle low, sample on falling edge)
Mode 2: CPOL=1, CPHA=0 (clock idle high, sample on falling edge)
Mode 3: CPOL=1, CPHA=1 (clock idle high, sample on rising edge)

Device datasheet specifies mode - must match!
```

**STM32 SPI Implementation:**
```c
void init_spi(void) {
    // SPI1 configuration
    // MOSI: PA7, MISO: PA6, SCK: PA5, CS: PA4
    
    #define SPI1_BASE 0x40013000
    
    volatile uint32_t *SPI_CR1 = (volatile uint32_t *)(SPI1_BASE + 0x00);
    volatile uint32_t *SPI_CR2 = (volatile uint32_t *)(SPI1_BASE + 0x04);
    
    // Configure:
    // BR: Baud rate prescaler (divide master clock)
    // MSTR: Master mode
    // CPOL, CPHA: Clock mode
    *SPI_CR1 = (0b001 << 3)  | // BR = /4
               (1 << 2)      | // MSTR (master)
               (0 << 1)      | // CPOL (idle low)
               (0 << 0);       // CPHA (sample on rising)
    
    *SPI_CR1 |= (1 << 6);       // Enable SPI
}

// Send and receive byte
uint8_t spi_txrx(uint8_t tx_byte) {
    volatile uint32_t *SPI_DR = (volatile uint32_t *)(0x40013000 + 0x08);
    volatile uint32_t *SPI_SR = (volatile uint32_t *)(0x40013000 + 0x00);
    
    while (!((*SPI_SR) & (1 << 1))) { }  // Wait for TX empty
    *SPI_DR = tx_byte;                   // Send
    
    while (!((*SPI_SR) & (1 << 0))) { }  // Wait for RX full
    return (uint8_t)(*SPI_DR);           // Receive
}

// Read sensor over SPI
uint16_t spi_read_sensor(void) {
    uint8_t high_byte, low_byte;
    
    GPIO_SET(GPIOA, 4);              // CS high (inactive)
    GPIO_CLEAR(GPIOA, 4);            // CS low (select device)
    
    high_byte = spi_txrx(0xFF);      // Send dummy, get high byte
    low_byte = spi_txrx(0xFF);       // Send dummy, get low byte
    
    GPIO_SET(GPIOA, 4);              // CS high (deselect)
    
    return ((uint16_t)high_byte << 8) | low_byte;
}
```

### Q3: I2C (Inter-Integrated Circuit / TWI)
**Answer:**
```
I2C Characteristics:
├─ Type: Synchronous, multi-master
├─ Speed: 100 kHz (standard), 400 kHz (fast), 3.4 MHz (high-speed)
├─ Wires: SDA (data), SCL (clock) - 2 wires only!
├─ Distance: Up to 10m with buffers
├─ Advantages: Fewer wires, multiple devices per bus
└─ Disadvantages: Lower speed, complex protocol

I2C Bus Structure (open-drain):
┌─────────────────┐
│  VDD (3.3V)     │
│  │              │
│  Rpu (pull-up)  │ Typical: 10kΩ external pull-ups
│  │              │
│  ├─── SDA ◄─────┼─── Master GPIO
│  │              │    Slave GPIO (open-drain)
│  │              │
│  ├─── SCL ◄─────┼─── Master GPIO
│  │              │    Slave GPIO (open-drain)
│
└─── GND
```

**I2C Transaction:**
```
START Condition: SDA goes low while SCL high
├─ Signals all devices to listen

Address + Read/Write Byte
├─ 7-bit address (0x00-0x7F)
├─ 1 bit R/W (0=write, 1=read)
├─ Slave with matching address pulls SDA low (ACK)

Data Bytes
├─ 8 bits of data
├─ Master sends, slave receives (write)
├─ OR slave sends, master receives (read)
├─ After each byte, receiver pulls SDA low (ACK)

STOP Condition: SDA goes high while SCL high
└─ Signals end of transaction
```

## Advanced Communication

### Q1: CAN Bus (Controller Area Network)
**Answer:**
```
CAN Characteristics:
├─ Type: Message-based, prioritized
├─ Speed: 500 kbps to 1 Mbps typical
├─ Wires: CAN_H, CAN_L, GND (differential)
├─ Distance: 100m at 500 kbps
├─ Bus arbitration: Bit-level (priority by message ID)
├─ Advantages: Reliable, prioritized, automotive standard
└─ Disadvantages: Complex, requires transceivers, expensive

CAN Arbitration (Priority):
- Lower ID = higher priority
- If multiple messages transmitted, lowest ID wins bus
- Other nodes back off automatically

CAN Message Structure:
┌───┬─────────┬────┬──────────┬──────┬─────┐
│ID │Standard │RTR │Reserved  │DLC  │Data │
│11 │1        │    │6         │4    │0-8  │
└───┴─────────┴────┴──────────┴──────┴─────┘

ID: 11-bit identifier (0-2047)
RTR: Remote transmission request
DLC: Data length (0-8 bytes)
```

### Q2: Ethernet and TCP/IP for embedded
**Answer:**
```
Ethernet in Embedded Systems:
├─ MCUs with Ethernet: STM32H7, Cortex-M7
├─ Speed: 100 Mbps (Fast Ethernet)
├─ Stack: lwIP (lightweight IP stack)
├─ Best for: IoT, industrial control, high-speed data

Typical architecture:
┌──────────┐
│ HTTP/TCP │ Application layer
├──────────┤
│  TCP/IP  │ lwIP stack (~50 KB)
├──────────┤
│ Ethernet │ MAC driver
└──────────┘

Benefits over serial:
- Much higher bandwidth
- Internet connectivity
- Web server capability
- Remote debugging

Drawback:
- Complex stack (ethernet transceiver, MAC driver)
- More power consumption
- Larger memory footprint
```

```
Communication Protocol Selection Matrix:

Need              │ UART  │ SPI   │ I2C   │ CAN   │ Ethernet
────────────────┼───────┼───────┼───────┼───────┼─────────

Multi-device     │       │       │ ✓✓✓   │ ✓✓    │ ✓✓✓
Automotive       │       │       │       │ ✓✓✓   │
Industrial       │       │       │       │ ✓✓    │ ✓✓
Internet         │       │       │       │       │ ✓✓✓
Battery device   │ ✓     │ ✓     │ ✓     │       │
────────────────┼───────┼───────┼───────┼───────┼─────────
Typical use:     │Serial │SPI    │I2C    │Vehic. │IoT
                 │term.  │Flash  │Sensors│       │

Decision tree:
1. Need internet? → Ethernet
2. Need priority arbitration? → CAN
3. Need multi-device on 2 wires? → I2C
4. Need high speed? → SPI
5. Simple, debug output? → UART
```
